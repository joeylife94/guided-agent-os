# Incident Summary Guidelines

## Incident Classification
Incidents are categorized by severity:
- **Critical**: System downtime, data loss risk, security breach
- **High**: Significant functionality impaired, escalated customer complaint
- **Medium**: Partial service degradation, requires investigation
- **Low**: Minor issues, workarounds available

## Summary Structure
1. **Incident Title**: Clear, concise one-line description
2. **Severity Level**: Use standardized classifications above
3. **Impact Area**: Which systems or services affected
4. **Root Cause**: If determined; mark "Under Investigation" if pending
5. **Resolution Status**: In Progress, Resolved, Escalated
6. **Timeline**: Incident start, detection, and resolution times

## Knowledge References
When documenting incidents, reference relevant domain knowledge:
- Operating procedures that were followed or violated
- Policy frameworks that apply to the incident type
- Previously resolved similar incidents for pattern matching

## Escalation Criteria
Escalate to human review when:
- Root cause cannot be determined automatically
- Resolution requires manual intervention
- Customer communication is needed
- Changes to core systems are required
